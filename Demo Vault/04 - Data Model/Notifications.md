# Notifications (Data Model)

> Laravel's built-in notifications table — a polymorphic log of every notification sent to every user.

---

## Concept Explained

The `notifications` table is created by Laravel's `php artisan notifications:table` migration. Each row stores a single notification as a JSON blob in the `data` column. The table uses `morphs('notifiable')` so it can theoretically attach notifications to any model (not just users). `read_at` being null means unread; set means read.

## How it's Used in Y

File: `database/migrations/2026_05_16_153926_create_notifications_table.php`

### Schema

```
notifications
├── id           UUID primary key (not auto-increment)
├── type         string — fully-qualified class name e.g. App\Notifications\LikeNotification
├── notifiable_type  string — polymorphic type (always "App\Models\User" in Y)
├── notifiable_id    bigint — the user receiving the notification
├── data         text — JSON blob from toArray()
├── read_at      timestamp nullable
└── timestamps
```

UUID primary key is important — it's generated by Laravel, not the database. This allows creating notification records in any order without collision.

### Notification types stored in Y

| `data.type` value | Class | When sent |
|---|---|---|
| `welcome` | `WelcomeNotification` | On registration |
| `post_created` | `PostCreatedNotification` | When user publishes a post |
| `like` | `LikeNotification` | Someone likes your post |
| `follow` | `FollowNotification` | Someone follows you |
| `reply` | `ReplyNotification` | Someone replies to your post |
| `profanity_strike` | `ProfanityStrikeNotification` | Strike issued |
| `ban` | `ProfanityStrikeNotification` (isBan=true) | Account banned |
| `password_updated` | `PasswordUpdateNotification` | Password changed |

### Querying unread count

```php
// In HandleInertiaRequests — runs on every request
$request->user()?->unreadNotifications()->count() ?? 0
```

`unreadNotifications()` is a scope provided by the `Notifiable` trait — it's equivalent to `->whereNull('read_at')`.

## Key Code Snippet

```php
// database/migrations/2026_05_16_153926_create_notifications_table.php
Schema::create('notifications', function (Blueprint $table) {
    $table->uuid('id')->primary();
    $table->string('type');
    $table->morphs('notifiable');   // adds notifiable_type + notifiable_id
    $table->text('data');
    $table->timestamp('read_at')->nullable();
    $table->timestamps();
});
```

---

## Why This Approach

Using Laravel's built-in database notification channel means no extra infrastructure — notifications are stored and read from the same database as everything else. The polymorphic design means the same table could later store notifications for other models (e.g. company accounts) without schema changes.
